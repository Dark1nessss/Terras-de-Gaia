## 1. How cPanel runs Node.js apps

### The full picture

```
Browser → Apache (port 80/443)
             ↓  (reverse proxy via .htaccess / Passenger)
         Phusion Passenger
             ↓  (spawns and supervises your process)
         node server.js   ← your app, on an internal port
             ↓
         Next.js request handler
```

cPanel does **not** run Node.js directly. It runs **Phusion Passenger**, a process manager, which in turn starts your app. Apache proxies all public HTTP traffic to Passenger.

### nodevenv — cPanel's Node.js version manager

When you select a Node.js version in cPanel's *"Setup Node.js App"* interface, it creates a **virtual environment** under `~/nodevenv/<appname>/`. This is similar to Python's virtualenv.

The important consequence: `node_modules` inside your app directory is actually a **symbolic link** pointing to the real location inside `~/nodevenv/`. Most apps work fine with this, but it breaks native binaries — like Next.js's native Rust compiler (`@next/swc`) — because the binary can't locate its own files through a symlink.

### What Passenger expects

Passenger looks for a **startup file** — by default `app.js`, but you can configure it. It runs `node <startup-file>` and expects your app to **listen on a TCP port** provided via the `PORT` environment variable. It does **not** use `npm start`; it calls Node directly.

## 2. Why we got so many errors

Here is a breakdown of every root cause, from most to least impactful.

### 2.1 `@next/swc` native binary not found → WASM fallback → Out of Memory

**What happened:**  
Next.js ships a native Rust-based compiler (`@next/swc-linux-x64-gnu`). When `node_modules` is a symlink (nodevenv), Node.js cannot resolve the binary's path relative to itself. Next.js silently falls back to a **pure WASM** implementation of the compiler.

The WASM compiler is 10–20× slower and uses far more memory. On a shared hosting account with a memory limit (typically 256–512 MB), the build or startup process simply gets killed by the OS.

**The fix (already in `server.ts` and `build.sh`):**  
Before loading Next.js, detect the symlink and add the real path to `NODE_PATH`, then call `Module._initPaths()`. This makes Node.js resolve native binaries through the real directory.

```ts
// server.ts — runs BEFORE next is required
const _nmPath = path.join(process.cwd(), 'node_modules');
if (lstatSync(_nmPath).isSymbolicLink()) {
  const realNm = realpathSync(_nmPath);
  process.env.NODE_PATH = realNm;
  Module._initPaths(); // re-scan paths so the fix takes effect immediately
}
```

### 2.2 `server.js` is empty — the compiled startup file was missing

**What happened:**  
`npm start` runs `node server.js`. However, `server.js` is generated by compiling `server.ts` with TypeScript. If that compilation step never ran, `server.js` is empty (or absent) and Node.js exits immediately with no useful error.

**The fix:**  
Add a TypeScript compile step for the server before running the app (see [Step 4](#4-step-by-step-deployment)).

```bash
npx tsc --project tsconfig.server.json
```

### 2.3 Build exceeded thread/memory limits on shared hosting

**What happened:**  
Next.js 15+ uses Turbopack/Tokio (Rust async runtime) internally. By default it spawns as many threads as there are CPU cores. On shared hosting, exceeding the OS thread limit causes the host to kill the process.

**The fix (already in `build.sh`):**

```bash
export TOKIO_WORKER_THREADS=2
export RAYON_NUM_THREADS=2
npm run build
```

### 2.4 `devDependencies` missing at build time

**What happened:**  
cPanel's Node.js App Manager sets `NODE_ENV=production` before running the install command, which causes `npm install` to skip `devDependencies`. This app needs TypeScript, Tailwind, and ESLint at build time — all of which live in `devDependencies`.

**The fix (already in `build.sh`):**

```bash
npm install --include=dev
```

### 2.5 Missing environment variables

**What happened:**  
The app requires `WORDPRESS_API_URL`, `WP_USER`, `WP_APP_PASSWORD`, and `INTERNAL_API_TOKEN` to be set. When they are missing, every page that fetches from the WordPress REST API fails during static generation, producing a cascade of build errors.

**The fix:** Set all variables before building and running (see [Section 5](#5-environment-variables)).

### 2.6 `.htaccess` passthrough not configured

**What happened:**  
Apache's `.htaccess` in the app's public root must instruct Passenger to handle all requests. Without this, Apache returns a 403 or serves static directory listings instead of the Node.js app.

**The fix:** See [Step 4.5](#step-45-configure-htaccess).

## 4. Step-by-step deployment

### Step 4.1 — Upload the source code

Upload the repository contents to your server, excluding `node_modules` and `.next`:

```bash
# From your local machine via rsync (recommended)
rsync -avz --exclude='node_modules' --exclude='.next' --exclude='.env*' \
  ./ user@yourhost.com:~/public_html/

# Or use cPanel File Manager / FTP — just don't upload node_modules
```

### Step 4.2 — Create the Node.js app in cPanel

1. Log in to cPanel → **Setup Node.js App**
2. Click **Create Application**
3. Fill in:

| Field | Value |
|---|---|
| **Node.js version** | `20.x` (or latest LTS) |
| **Application mode** | `Production` |
| **Application root** | Path to your app, e.g. `public_html` |
| **Application URL** | Your domain or subdomain |
| **Application startup file** | `server.js` |

4. Click **Create**. cPanel will generate the nodevenv virtual environment.
5. **Copy the activation command** shown on screen — you need it for SSH steps.

### Step 4.3 — Set environment variables in cPanel

Still on the *Setup Node.js App* page, scroll down to **Environment Variables** and add:

| Name | Value |
|---|---|
| `NODE_ENV` | `production` |
| `WORDPRESS_API_URL` | `https://terrasdegaia.pt/wp-json` |
| `WP_USER` | your WordPress username |
| `WP_APP_PASSWORD` | your WordPress application password |
| `INTERNAL_API_TOKEN` | a 32-character random string |
| `PORT` | leave blank (Passenger sets this automatically) |

Click **Save**.

### Step 4.4 — Build the app via SSH

SSH into your server and run:

```bash
# Enter the app directory
cd ~/public_html

# Activate the nodevenv virtual environment (use the command cPanel showed you)
source ~/nodevenv/public_html/20/bin/activate

# Verify you're using the right Node version
node -v   # should print v20.x.x
npm -v

# Run the build script (handles nodevenv quirks + thread limits)
bash build.sh

# Compile server.ts → server.js
npx tsc --project tsconfig.server.json

# Verify server.js was created and is not empty
wc -l server.js   # should be > 5 lines
```

> **Why `bash build.sh` instead of `npm run build`?**  
> `build.sh` sets `NODE_PATH` for the symlinked `node_modules`, limits thread counts, and forces `--include=dev` on install. Running `npm run build` directly skips all of that.

### Step 4.5 — Configure `.htaccess`

Create or update `~/public_html/.htaccess` (the public-facing root Apache uses):

```apache
# Tell Apache to hand everything to Passenger (your Node.js app)
PassengerEnabled on
PassengerNodejs /home/<your_cpanel_user>/nodevenv/public_html/20/bin/node

# Passenger looks for server.js by default; confirm startup file
PassengerStartupFile server.js
PassengerAppEnv production

# Remove any existing index file directives that might conflict
DirectoryIndex disabled

# Pass all requests through
RewriteEngine On
RewriteRule ^(.*)$ - [L]
```

Replace `<your_cpanel_user>` with your actual cPanel username.

### Step 4.6 — Start the app

Back in cPanel → **Setup Node.js App**, click **Run NPM Install** (this installs inside the nodevenv), then click **Start Application** (or Restart if already running).

### Step 4.7 — Verify

```bash
# Check the process is running
ps aux | grep node

# Check the app's log
tail -f ~/logs/public_html.error.log

# Test from the server itself
curl -I http://localhost:$(grep PORT ~/nodevenv/public_html/.env 2>/dev/null || echo 3000)
```

## 5. Environment variables

All variables must be set **before** building. The safest place is the cPanel UI (Section 4.3) because it writes them into the nodevenv activation script and Passenger picks them up automatically.

You can also create a `.env.production.local` file in the app root for server-side variables (it is git-ignored):

```bash
# .env.production.local  — DO NOT commit this file
NODE_ENV=production
WORDPRESS_API_URL=https://terrasdegaia.pt/wp-json
WP_USER=terras_de_gaia
WP_APP_PASSWORD=xxxx xxxx xxxx xxxx xxxx xxxx
INTERNAL_API_TOKEN=your_32_character_random_string_here
```

> **Important:** `NEXT_PUBLIC_*` variables are baked into the JavaScript bundle at **build time**. If you change them you must rebuild (`bash build.sh`). Server-side-only variables (like `WP_APP_PASSWORD`) are read at runtime — change them in cPanel UI and restart without rebuilding.

## 6. Keeping the app alive

Passenger restarts your app automatically if it crashes. However, on shared hosting Passenger may be configured to stop idle apps after a period of inactivity to free memory. To keep the app warm you can:

- Set up a cron job that pings your domain every 5 minutes:

  ```
  */5 * * * * curl -s https://terrasdegaia.pt/ > /dev/null
  ```

  Add this in cPanel → **Cron Jobs**.

- Ask your host to increase the Passenger idle-timeout for your app.

## 7. Common errors and fixes

### `Error: Cannot find module '@next/swc-linux-x64-gnu'`

**Cause:** nodevenv symlink breaks native binary resolution.  
**Fix:** Make sure `server.js` was compiled from `server.ts` (which contains the `NODE_PATH` fix). Verify with `head -5 server.js` — you should see real code, not a blank file.

---

### `FATAL ERROR: Reached heap limit — Allocation failed`

**Cause:** Next.js fell back to WASM compiler, or the build process exceeded the server's memory limit.  
**Fix:**
1. Ensure `server.ts` is compiled correctly (the symlink fix must run first).
2. Set memory limits in the build step:
   ```bash
   export NODE_OPTIONS="--max-old-space-size=512"
   bash build.sh
   ```
3. If still failing, contact your host to increase the memory limit for your account.

---

### `Error: EACCES: permission denied` on `node_modules/.bin/*`

**Cause:** npm tried to install into the real nodevenv path but lacks write permission.  
**Fix:** Always run `npm install` **after** activating the nodevenv environment (`source ~/nodevenv/.../activate`), not as root.

---

### App returns `503 Service Unavailable` immediately after start

**Cause:** `server.js` is empty or has a syntax error, so Node exits right away. Passenger reports 503.  
**Fix:**
```bash
node server.js   # run manually to see the real error
```
If blank, recompile: `npx tsc --project tsconfig.server.json`.

---

### Pages load but images return `400 Bad Request`

**Cause:** `next/image` optimisation is trying to process images but the `/_next/image` route isn't being proxied correctly.  
**Fix:** Ensure the `.htaccess` is not intercepting `/_next/` paths. Add this **before** other rewrite rules:
```apache
RewriteRule ^_next/(.*)$ - [L]
```

---

### Build fails with `[AUTH] Missing WP_USER or WP_APP_PASSWORD`

**Cause:** Environment variables are not set before `npm run build`.  
**Fix:** Set them in cPanel UI first (Section 4.3), then SSH in and activate the nodevenv before building — the activation script exports those variables into the shell.

---

### `TOKIO_WORKER_THREADS` / thread limit errors

**Cause:** Next.js's Rust runtime tries to spawn more threads than the host allows.  
**Fix:** Already handled in `build.sh`. If you run the build outside that script, set manually:
```bash
export TOKIO_WORKER_THREADS=2
export RAYON_NUM_THREADS=2
```

## 8. Updating the app

When you push new code:

```bash
# 1. Upload changed files (exclude node_modules and .next)
rsync -avz --exclude='node_modules' --exclude='.next' --exclude='.env*' \
  ./ user@yourhost.com:~/public_html/

# 2. SSH in and activate nodevenv
cd ~/public_html
source ~/nodevenv/public_html/20/bin/activate

# 3. Rebuild
bash build.sh
npx tsc --project tsconfig.server.json

# 4. Restart the app via cPanel UI (Setup Node.js App → Restart)
# or via command line:
touch tmp/restart.txt   # Passenger will restart on next request
```

> **Never** run `npm start` manually in SSH to "test" in production — that starts a second Node process on the same port, causing conflicts. Always use the cPanel UI Start/Restart button or the `tmp/restart.txt` touch trick.
